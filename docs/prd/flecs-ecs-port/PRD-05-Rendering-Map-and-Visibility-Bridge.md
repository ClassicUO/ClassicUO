# PRD-05: Rendering, Map, and Visibility Bridge

## 1. Objective

Feed the existing rendering stack from ECS-derived data while preserving visual correctness, layering, and map behavior.

## 2. Scope

In scope:

- Render extraction from ECS
- Visibility/range/map chunk integration
- Draw ordering migration strategy
- World-map and minimap data feeds

Out of scope:

- Replacing FNA rendering backend
- Complete UI redraw architecture change

## 3. Current State

- Draw/update logic is scene-centric and reads mutable world objects directly (`GameScene`, sorting helpers, render lists).
- Visibility, lighting, and object filtering rely on entity methods/managers and map chunk iteration.

## 4. Target Architecture

Keep renderer backend, change state source:

- ECS systems populate frame-local `RenderExtract` buffers/components.
- Existing render code consumes extracted data views instead of traversing mutable game objects.
- Maintain render-thread assumptions (main thread execution unless explicitly refactored).

## 5. Render Data Model

Introduce extraction components/records:

- `RenderSprite`
- `RenderAnimationFrame`
- `RenderLayerKey`
- `RenderLightContribution`
- `RenderNameplate`
- `RenderSelectionFlags`

Guideline:

- Render extract structs contain primitive/ID data only.
- Any heavyweight asset references resolved late by renderer using IDs.

## 6. Visibility and Map Integration

Map strategy:

- Represent map/chunk scope metadata in ECS (chunk entity + tags).
- Entities reference map/chunk IDs via components/relations.
- Visibility systems compute in-range and in-frustum tags.

Behavior parity targets:

- Existing `ClientViewRange` semantics
- House/multi visibility quirks
- Seasonal graphic changes and light behavior

## 7. Draw Order and Sorting

Use query sorting/grouping where manual lists are now maintained:

- Group by map/chunk or Z-bands when beneficial
- Secondary sort by draw keys matching current scene ordering rules
- Keep exact tie-breaking behavior where gameplay readability depends on it

## 8. Lighting and Effects

- Light sources become entities/components.
- Lighting extraction computes per-frame light inputs for renderer.
- Effects retain lifetimes in ECS and emit render contributions until expiry.

## 9. World Map and Overlay Feeds

World map bridge:

- ECS query provides mobile/item/party marker states.
- `WorldMapGump` integration reads through adapter instead of direct manager dictionaries.

Nameplates/overheads:

- Text emit requests generated by ECS events and consumed by UI bridge.

## 10. Migration Tasks

1. Add render extraction module with no-op outputs.
2. Migrate static item/mobile sprite extraction.
3. Migrate ordering/layer key generation.
4. Migrate lighting/effect extraction.
5. Migrate world-map/minimap feeds.
6. Remove direct world traversal from migrated draw paths in ECS mode.

## 11. Performance Targets

- No regression in average FPS relative to legacy mode in baseline scenes.
- No >10% regression in 95th percentile frame time in stress scenes.
- Allocation budget: no sustained per-frame managed allocation growth in hot render extract path.

## 12. Acceptance Criteria

- Visual parity for core scenes (movement, combat, containers, houses, world map).
- Draw ordering and culling behavior pass parity screenshots/checklists.
- Renderer reads ECS-backed extraction data for migrated paths in ECS mode.

## 13. Risks and Mitigations

Risk: subtle ordering regressions
- Mitigation: golden screenshot packs + deterministic camera paths for diffing

Risk: over-coupling extraction with renderer internals
- Mitigation: strict boundary; extraction outputs neutral render records

Risk: map edge-case behavior loss
- Mitigation: explicit map/house/world-map parity test suites
